Contention Pattern

                        
48K: Screen lines use 192 T-states for drawing. Memory contention occurs when CPU accesses 4000-7FFF during screen drawing (lines 64-255, T-states 14-126).

                        
128K/+2: Screen lines use 196 T-states for drawing (228 total per line). Contended banks: 1, 3, 5, 7 (odd).

                        
+2A: Same timing as 128K. Contended banks: 4, 5, 6, 7 (high). In special paging mode, contention applies per-slot based on the mapped bank.

                        
Pattern repeats every 8 T-states: 6, 5, 4, 3, 2, 1, 0, 0 (delay in T-states)

                        
I/O contention: ULA port (xxFE) also causes delays when accessed during screen time.

                    


                    

                        
+2A Differences (vs 128K)

                        
    
                            
  • 4 ROM banks (64KB) selected via ports 0x7FFD bit 4 + 0x1FFD bit 2
    • 
                            
  • Port 0x1FFD: special paging, ROM high bit, disk motor
    • 
                            
  • 4 special all-RAM paging modes (port 0x1FFD bit 0 = 1)
    • 
                            
  • Different contended banks: 4, 5, 6, 7 (not 1, 3, 5, 7)
    • 
                            
  • 48K BASIC ROM in bank 3 (not bank 1)
    • 
                        

                    


                    

                        
Pentagon Differences

                        
    
                            
  • No memory or I/O contention
    • 
                            
  • 320 lines per frame (more top border)
    • 
                            
  • TR-DOS ROM pages in/out via 3Dxx trigger
    • 
                            
  • Beta Disk interface at ports 1F, 3F, 5F, 7F, FF
    • 
                        

                    


                    

                        
Scorpion ZS 256 Differences

                        
    
                            
  • 256KB RAM (16 pages), Pentagon-compatible ULA timing
    • 
                            
  • 4 ROM banks: ROM0=128 BASIC, ROM1=48 BASIC, ROM2=Service Monitor, ROM3=TR-DOS
    • 
                            
  • Port 0x7FFD: standard 128K paging (RAM 0-7, ROM 0/1, screen, lock)
    • 
                            
  • Port 0x1FFD: bit 0 = RAM over ROM, bit 1 = ROM 2 select, bit 4 = RAM page +8
    • 
                            
  • ROM selection (3-way): 1FFD.1 set → ROM 2; unset → 7FFD.4 selects ROM 0/1
    • 
                            
  • TR-DOS built into ROM bank 3 (no separate trdos.rom needed)
    • 
                            
  • No memory or I/O contention (same as Pentagon)
    •

Overview

                    
ZX-M8XXX (ZX Matrix) is a vanilla JavaScript ZX Spectrum emulator with integrated debugger for reverse engineering.

                    
Features

                    
    
                        
  • Full Z80 CPU emulation (all documented + undocumented opcodes)
    • 
                        
  • 48K, 128K, +2, +2A, +3, Pentagon, and Scorpion machine support
    • 
                        
  • Memory banking (128K/+2/+2A/Pentagon/Scorpion)
    • 
                        
  • ULA video generation with border
    • 
                        
  • ULAplus extended palette (64 colors, HAM256 raster effects)
    • 
                        
  • Memory contention emulation
    • 
                        
  • AY-3-8910 sound chip (stereo modes)
    • 
                        
  • PSG file export (record AY music)
    • 
                        
  • SNA/Z80/SZX snapshot loading/saving
    • 
                        
  • Quicksave/Quickload (F2/F5) - instant state save to browser storage
    • 
                        
  • TAP/TZX tape loading (instant or real-time with border stripes and sound)
    • 
                        
  • TRD/SCL disk images (TR-DOS)
    • 
                        
  • DSK disk images (+3 µPD765 FDC)
    • 
                        
  • ZIP archive support
    • 
                        
  • RZX playback (partial)
    • 
                        
  • Game Browser - search and download games from Spectrum Computing
    • 
                        
  • Fullscreen mode (F11) with crisp/fit/stretch options
    • 
                        
  • Automated test suite with screenshot comparison
    • 
                        
  • Compare tool for snapshots and binaries
    • 
                        
  • Explorer for file analysis (BASIC decoder, disassembly, hex dump)
    • 
                        
  • Game Mapper for capturing rooms and building navigable map grids
    •

Signature Packs

                    
Modular knowledge base for automatic label and region recognition. Each pack is a self-contained JSON file with labels, regions, and byte-pattern anchors for matching.

                    
Pack Types

                    
    
                        
  • Engine packs — Match known engines (AGD, Quill, PAW, etc.). One pack covers all games using that engine version.
    • 
                        
  • Game packs — Labels for a specific game from community disassemblies.
    • 
                    

                    
Workflow

                    
    
                        
  1. Import a .skool file, .asm/.a80 source, .zip archive, or pack JSON via Settings → Signatures
    2. 
                        
  2. Build Anchors — with the game loaded, click "Anchors" to snapshot byte patterns at key labeled addresses. Anchors automatically mask 16-bit address operands (CALL nn, JP nn, LD rr,nn, etc.) for cross-game matching.
    3. 
                        
  3. Scan Memory — load a game and click "Scan Memory" to find matching packs. Results show confidence percentage. Two-pass matching: exact first, then masked fallback.
    4. 
                        
  4. Apply — click "Apply" on a match to add labels and regions. Per-anchor offset voting handles procedures relocated to different addresses.
    5. 
                        
  5. Direct Apply — click "Apply" in the pack list to apply all labels at offset 0 (no scanning needed).
    6. 
                    

                    
Cross-Game Matching

                    
Signature packs can match code across different games sharing the same engine (e.g. applying EATF labels to Bomberman). Each procedure is matched independently with its own offset, so shuffled code is handled correctly. Address operands in Z80 instructions are masked automatically, since absolute addresses differ between builds.

                    
Supported Formats

                    
    
                        
  • .skool (SkoolKit) — reads @label= directives, control chars (c=code, b=data, t=text, w=words), inline comments
    • 
                        
  • .asm / .a80 (sjasmplus, pasmo, z80asm) — reads labels, ORG directives, EQU constants, data directives (DEFB/DEFW/DEFS/DEFM). INCLUDE directives resolved when importing multiple files.
    • 
                        
  • .zip — archives containing any combination of .skool, .asm, and .a80 files. All source files are extracted and processed together.
    •

POKE Search

                    
Find memory locations storing game values (lives, score, etc.) by comparing snapshots:

                    
    
                        
  1. Snap — take a memory snapshot (status shows snap count)
    2. 
                        
  2. Play game until the target value changes (e.g. lose a life)
    3. 
                        
  3. Snap again — repeat snap/play for more data points
    4. 
                        
  4. Search — scans all RAM, validates every snap-to-snap transition:
                            
      
                                
    • -1 / +1 — value changed by exactly 1 between each snap
      • 
                                
    • Decreased / Increased — value went down/up between each snap
      • 
                                
    • Changed — value differs between each snap
      • 
                                
    • Unchanged — value stayed the same across all snaps
      • 
                            
    
                        
    5. 
                        
  5. Take more snaps and search again to narrow candidates
    6. 
                        
  6. Filter — enter a hex value to keep only candidates where current memory matches. Clear the value and click Filter to undo
    7. 
                        
  7. Click result to view in memory dump; hover to see value at every snap point
    8. 
                    

                    
Requires at least 2 snapshots. Each search always scans all RAM — more snaps = fewer candidates. No deduplication: every snap-to-snap pair must match the selected mode.

                    
Skip screen (on by default) — excludes screen memory (4000-5BFF) to reduce false positives.

Game Mapper

                    
The Mapper tab (Tools → Mapper) captures game screens and stitches them into a navigable map grid. Useful for documenting game layouts, creating maps, and reverse engineering room structures.

                    
Basic Workflow

                    
    
                        
  1. Navigate to a room in the game
    2. 
                        
  2. Click Capture (or Ctrl+Space) to capture the screen
    3. 
                        
  3. Use arrow buttons (or Ctrl+Arrow keys) to move to an adjacent grid position
    4. 
                        
  4. Navigate to the next room in the game and capture again
    5. 
                        
  5. Repeat until the map is complete
    6.

  • Room label — shows current (X, Y) grid coordinates
    • 
                        
  • Floor buttons (F: ▼ / ▲) — navigate between floors for multi-level games
    • 
                        
  • Blend — per-pixel mode blend of all screenshots (picks most frequent color at each pixel, extracting static background and discarding moving sprites). Appears when room has 2+ screenshots
    • 
                        
  • Stamp — copy a rectangular region from the selected screenshot onto the blended image to correct blend artifacts (appears when blend exists and a screenshot is selected). Opens a dialog over the overview — drag to select a rectangle, pixels are stamped immediately. Multiple stamps supported; Escape to close

  • Mark color — mark the room with a colored diagonal cross (X) overlay using one of the 8 ZX Spectrum colors. Marks appear in the overview, hover popup, and PNG export. Select “No mark” to remove
    • 
                        
  • Delete — delete the currently selected screenshot
    • 
                        
  • Thumbnails — clickable strip of all screenshots for the current room; click to select which is displayed
    • 
                        
  • ⚙ (gear) — toggle the settings panel

    • 
                        
  • Highlight — color used to outline the current room in the overview

  • Gap H / V — horizontal and vertical pixel spacing between rooms in the overview and export (useful for games with non-connected rooms or varying row/column spacing)

POKE Manager

                    
The Pokes tab in the debugger panel manages named pokes (multi-address byte patches) and memory value editors. Two-column layout: editors on the left, pokes on the right. Create entries inline or load from JSON files.

                    
Layout

                    
Editors column (left) and pokes column (right) are shown side by side. Load, Save, and Clear buttons are at the top right. Click the game name label to edit it.

                    
Adding Pokes

                    
Use the add form at the top of the pokes column: enter a name, address, original value, and poke value, then click +. If a poke with the same name already exists, the new patch is added to it (multi-patch building). Addresses and values accept $hex, 0xhex, or decimal.

                    
Poke Entries

                    
Each poke has a checkbox to toggle it on/off. Enabling writes the patched bytes to memory; disabling restores the original values. Changes are instant — no apply button needed. The master checkbox above the poke list enables or disables all pokes at once (indeterminate state when partially enabled). Click × to remove an entry (restores original bytes if enabled).

                    
Adding Editors

                    
Use the editor add form at the top of the editors column: enter a name, address, and value type (byte or word), then click +. Read refreshes all editor values from memory.

                    
Memory Editors

                    
Editors provide live read/write access to specific memory addresses. Values accept $hex, 0xhex, or decimal input; displayed with $ prefix. Editing a value and pressing Enter or leaving the field writes it to memory immediately. Click × to remove an editor.

                    
Word editors use little-endian byte order (low byte at address, high byte at address+1).

                    
JSON File Format

                    
Create poke files in a text editor. Addresses and values accept $hex, 0xhex, or decimal formats. Patches use compact [addr, normal, poke] arrays.

                    
{
    "game": "Game Name",
    "pokes": [
        {
            "name": "Infinite lives",
            "enabled": true,
            "patches": [["$8C2A", "$3D", "$00"]]
        },
        {
            "name": "Walk through walls",
            "patches": [["$8E01", "$28", "$18"], ["$8E10", "$3C", "$00"]]
        }
    ],
    "editors": [
        { "name": "Lives", "addr": "$8C2B", "type": "byte" },
        { "name": "Score", "addr": "$8C30", "type": "word" }
    ]
}

                    
Fields

                    
    
                        
  • game — optional display label shown in the toolbar
    • 
                        
  • pokes[].name — display name for the poke entry
    • 
                        
  • pokes[].enabled — initial enabled state (boolean, default false)
    • 
                        
  • pokes[].patches[] — arrays of [addr, normal, poke]
    • 
                        
  • editors[].name — display label for the editor
    • 
                        
  • editors[].addr — memory address
    • 
                        
  • editors[].type — "byte" or "word" (little-endian)
    • 
                    

                    
Project Persistence

                    
Pokes and editors are saved and restored with the project file, including which pokes are currently enabled.

  • Floating bus: Not fully emulated. Some copy protection schemes may not work.
    •
